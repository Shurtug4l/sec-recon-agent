# ISO/IEC 42001:2023 alignment

This document maps `sec-recon-agent` against [ISO/IEC 42001:2023](https://www.iso.org/standard/81230.html) - the management-system standard for AI. ISO/IEC 42001 is process-heavy by design: it specifies what an organization running an AI Management System (AIMS) must demonstrate, not how a specific codebase must look. Most clauses apply to an organization, not a single repository.

What this document **is**:

- A row-per-clause / per-control map of where this codebase already provides evidence that an AIMS would point at.
- A statement of which clauses are deliberately out of scope for a single-author portfolio project (no procurement process, no governance committee, no incident-response runbook beyond `SECURITY.md`).
- An honest declaration: this repo is not certified against ISO/IEC 42001 and does not claim to be. It demonstrates that the technical artefacts the standard expects can be produced.

What this document is **not**:

- A certification audit.
- A management-system manual.
- A substitute for the organizational processes ISO/IEC 42001 requires.

## Detailed mappings

### Clause 4 - Context of the organization

The organizational context an AIMS would document includes purpose, scope, stakeholders, and interested-party requirements. For a portfolio project the relevant artefacts are:

- **Purpose**: `README.md` opening paragraph - type-safe security triage on top of Pydantic AI and a custom MCP server, with the threat model the project takes as its design constraint.
- **Scope**: `docs/design.md::What this is` plus the explicit residual-risks section that declares what is intentionally out of scope (single-tenant demo, no procurement workflow).
- **Stakeholders / interested parties**: implicit (operator, end user, reviewer). For a multi-tenant deployment this section would expand to include data subjects and regulators.

### Clause 6 - Planning (risk management)

The AIMS risk-management plan is the documented loop of identify -> assess -> treat -> monitor for AI risks. Evidence in this repo:

- **Identify + assess**: `docs/design.md::Threat model` with finding-to-fix mapping (independent security review). `docs/owasp_llm_top10.md` and `docs/mitre_atlas.md` map the threat space against external taxonomies.
- **Treat**: every "HIGH" review finding is tied to a code change. Layered controls per risk are listed inline in the mapping docs.
- **Monitor**: `sec-recon-redteam` provides a regression detector against prompt-injection resistance after a system-prompt or model change. `sec-recon-eval` does the same for general grounding correctness.

### Annex A.2 - AI policy

In an AIMS, A.2 is the documented AI policy approved by leadership. The equivalent in this repo is the machine-readable policy that constrains every model output:

- **System prompt**: `src/sec_recon_agent/agent/prompts.py`. Names every tool the agent may call, declares the untrusted-content boundary, and pins the structured output contract.
- **Public posture**: `SECURITY.md` declares responsible disclosure, in-scope and out-of-scope, safe-harbor.

A.2 in a real AIMS would also include sign-off, review cycle, and distribution evidence. For a single-author project the system prompt + SECURITY.md is the closest analog; sign-off corresponds to the commit history.

### Annex A.5 - AI system impact assessment

A.5 requires a documented assessment of the impact of AI use on individuals, groups, and society. Evidence:

- `docs/owasp_llm_top10.md` - per-risk impact assessment against the OWASP taxonomy, with the layered control list and the test that defends the invariant.
- `docs/mitre_atlas.md` - adversary-side impact assessment against the ATLAS framework.
- `docs/design.md::Threat model` - actor-centric impact analysis with the controls mapped to specific findings.
- `src/sec_recon_agent/redteam/payloads.py` - the operational signal that the impact-mitigation controls still hold after a prompt or model change.

### Annex A.6 - AI system life cycle

The standard expects evidence across data acquisition, model selection, validation, deployment, and decommissioning.

- **Data acquisition**: `docs/design.md::Decisions log` records why NVD + CISA KEV + FIRST EPSS + Exploit-DB + GitHub + bundled MITRE ATT&CK are the sources (public, deterministic, signed-payload tradeoffs documented).
- **Model selection**: per-request `model` override + `ALLOWED_MODELS` allowlist in `agent/triage.py`. `make eval-compare` produces side-by-side metrics across `haiku` / `sonnet` / `opus`.
- **Validation**: 312 fast tests + 2 slow + soft-assertion eval against an 11-case golden set + an 18-payload red-team battery. Coverage ~89% with a 70% floor.
- **Deployment**: Docker Compose stack with hardened containers (non-root, read-only filesystem, no-new-privileges, `127.0.0.1` bindings).
- **Decommissioning**: not applicable to a portfolio project. The audit trail (`audit/`) would survive decommissioning; the rest of the runtime state lives in named Docker volumes the operator controls.

### Annex A.7 - Data for AI systems

Data governance + minimization + retention.

- **Source provenance**: every tool documents its upstream source in its docstring; the audit row records the source by recording the tool call.
- **Minimization**: queries are SHA-256 hashed by default, never persisted as plain text unless `AUDIT_INCLUDE_QUERY=true`. ChromaDB corpus excludes user queries.
- **Retention**: append-only log on local disk. Operator-controlled.
- **Untrusted-content fencing**: every free-text vendor field reaches the LLM inside `<UNTRUSTED_CONTENT>` markers. Span attributes never carry vendor text. Canary tests defend the invariant.

### Annex A.8 - Information for interested parties

Transparency + explainability + incident communication.

- **Transparency**: `GET /v1/meta` returns the literal system prompt and the tool inventory. Frontend `/dashboard` renders both.
- **Explainability**: every TriageReport carries a `reasoning_chain` field - ordered audit log of tool calls and decisions. `confidence` constrains over-claiming.
- **Incident communication**: `SECURITY.md` with responsible disclosure, coordinated window, in-scope / out-of-scope, safe harbor.

### Annex A.9 - Use of AI systems

A.9 requires controls on use-time behavior: guardrails, monitoring, override mechanisms.

- **Guardrails**: structured Pydantic output prevents free-text emission outside `TriageReport`. `ALLOWED_MODELS` allowlist prevents probing arbitrary model identifiers via the body field.
- **Monitoring**: OTel spans on every tool call with stable attribute whitelist; `sec-recon-audit verify` provides tamper-evident integrity checking.
- **Override**: opt-in API authentication (`API_KEYS`) + per-IP rate limit (`RATE_LIMIT_PER_MINUTE`) give the operator request-level control. The agent itself has no out-of-band tools - every action is a typed MCP call.

### Annex A.10 - Third-party relationships

Supplier risk and customer responsibilities.

- **Upstream sources documented**: NVD (NIST), CISA KEV, FIRST EPSS, Exploit-DB (GitLab), GitHub, MITRE ATT&CK. Each tool docstring names the source + the reliability assumptions made.
- **Supply chain hardening**: pinned `uv.lock` + `frontend/package-lock.json`; Dependabot weekly on pip, npm, docker, github-actions; host-locked redirects on every external fetch; Dockerfile uses official base images with `apt upgrade` + Trivy / docker-scout validation.
- **Customer-side responsibilities**: documented in `SECURITY.md` (use of the demo behind a public network requires API key + rate limit + audit + isolated container) and `docs/design.md::Residual risks`.

## Clauses deliberately out of scope

These clauses require organizational processes that a single-author repository cannot legitimately claim:

- **Clause 5 (Leadership)**: no AI policy committee, no top-management commitment artefact beyond the commit history.
- **Clause 7.2 (Competence)**: no training records, no role definitions.
- **Annex A.3 (Internal organization)**: no team to allocate AI responsibilities across.
- **Annex A.4 (Resources)**: no resource-allocation process.

A real AIMS certification audit would weigh evidence under each of these clauses heavily. This repo demonstrates that the **technical artefacts** the standard depends on can be produced in code; the **process artefacts** would be produced by the deploying organization.
